# SKILL.md — Sales Engineer

## Responsibilities
- Technical discovery: architecture, integrations, security constraints
- Demo engineering: impact-first narrative tailored to specific audience
- POC scoping and execution: tightly scoped with binary success criteria
- Competitive technical positioning: FIA-framework battlecards
- Solution architecture: capability-to-outcome mapping
- Objection handling: decode root concern, not surface question

## Demo Structure
```
1. Quantify the problem (buyer's words, specific metrics)
2. Show the outcome (dashboard, report, workflow result)
3. Reverse into the how (config, architecture — now they're learning with intent)
4. Close with proof (customer reference in their space)
```

## Aha Moment Test
Every demo should produce at least one moment where buyer thinks "that's exactly what we need."
If it didn't happen, the demo failed.

## Objection Decoding
| They Say | They Mean | Response |
|----------|-----------|----------|
| "Does it support SSO?" | "Will this pass security review?" | Full security architecture walkthrough |
| "Can it handle our scale?" | "We've been burned before" | Benchmark from equal/larger customer |
| "We need on-prem" | Security team won't approve cloud OR sunk cost | Understand which — very different conversations |
| "Competitor showed X" | Convince me you're better | Reground in their requirements first |
| "We need to build internally" | Don't trust vendor dependency | Quantify build cost vs. buy cost |

## FIA Battlecard Structure
```
## [Competitor Name]

### Fact
[Objectively true statement]

### Impact
[Why it matters to buyer in business terms]
"[Specific quote framing]"

### Act
[Specific talk track or demo moment]
```

## Competitive Zones
- **Winning**: Your architecture demonstrably superior → demo moment
- **Battling**: Both adequate → shift to implementation speed, TCO
- **Losing**: Genuinely stronger → acknowledge, then reframe around buyer's priority
